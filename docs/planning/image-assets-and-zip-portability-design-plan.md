# Image Assets And ZIP Portability Design Plan

Date: July 9, 2026  
Perspective: Staff software engineer for web, mobile, and frontend systems

## Objective

Add images to Valgaron records from two sources:

- a user-provided web image URI, stored as a URI string in the world document;
- a user-uploaded image, represented by a URI string plus durable asset metadata
  in an expanded world document, while its binary bytes live in a platform-owned
  local asset store.

Add a ZIP backup that contains the same generated JSON backup, unchanged, and
the uploaded image files referenced by that JSON. Remote web images remain URI
references and are not downloaded into the ZIP.

The design remains local-first and web-first. It must not imply cloud backup,
remote-image availability, collaboration, or cross-device synchronization.

## Current Runtime Findings

The plan extends these existing seams:

- `WorldDocument` is schema version `3` and is validated centrally in
  `packages/core/src/worldDocument.ts`.
- Every codex record shares `WorldEntryBase`; it currently has no image field.
- Web persists the document as one JSON string in `localStorage` under
  `valgaron.worldDocument.v3`.
- Mobile persists the same shared document shape as one JSON string in
  `AsyncStorage` under `valgaron.mobile.worldDocument.v3`.
- Core already produces active-workspace and full-document JSON backups. Web
  downloads those strings; mobile shares them as text.
- JSON import replaces the current document after shared parsing, validation,
  preview, and destructive-action confirmation.
- Recovery snapshots contain a complete `WorldDocument` but no separate binary
  payload.
- The exported `ImageUploader` creates temporary `blob:` object URLs only. It
  is not connected to codex records, and those URLs are invalid after reload.
- `ImageView` can display URI-backed images, but it is not integrated into entry
  detail or editor workflows.
- There is no application ZIP dependency, IndexedDB asset repository, or mobile
  app-managed file repository today.

Consequently, neither base64 strings in the JSON nor temporary `blob:` URLs are
appropriate. Base64 would inflate and eventually exceed string-storage limits;
`blob:` URLs are process-local handles rather than durable identifiers.

## Proposed Architecture At A Glance

```text
Entry image reference
        |
        | uri: https://...            -> browser/native remote image loader
        |
        ` uri: images/<asset-id>.png   -> uploaded asset catalog lookup
                                              |
                       +----------------------+----------------------+
                       |                                             |
                 Web IndexedDB                              Mobile app files
                 stores Blob bytes                         store binary bytes
                       |                                             |
                       +----------------------+----------------------+
                                              |
                                        ZIP backup builder
                                        JSON + images/*
```

Shared core owns the schema, reference validation, asset reachability, backup
selection, and ZIP package description. Each platform owns binary selection,
storage, URI resolution, ZIP encoding/download or share-sheet delivery, and
object URL/file URI lifecycle.

## Recommended Data Contract

This is the recommended contract, subject to the decision gates below.

```ts
export type WorldImageReference = {
  id: string;
  uri: string;
  altText: string;
  caption: string;
};

export type WorldImageAsset = {
  id: string;
  uri: string;
  originalFilename: string;
  mediaType: 'image/jpeg' | 'image/png' | 'image/webp' | 'image/gif';
  byteSize: number;
  sha256: string;
  createdAt: string;
};

export type WorldEntryBase = {
  // Existing fields remain unchanged.
  images: WorldImageReference[];
};

export type WorldDocument = {
  schemaVersion: 4;
  activeWorldId: string;
  worlds: WorldWorkspace[];
  assets: WorldImageAsset[];
  savedAt: string;
};
```

### URI Rules

- Remote image: `https://images.example.test/map.png`.
- Uploaded image: `images/asset-<generated-id>.png`.
- The entry reference always contains a `uri` string.
- An uploaded URI must have exactly one matching `WorldImageAsset` row.
- A remote URI must not have an asset row.
- `blob:`, `data:`, `file:`, `javascript:`, protocol-relative, and arbitrary
  relative URIs are invalid persisted sources.
- Uploaded paths are generated by the app. User filenames are metadata only and
  never become ZIP paths or storage keys.

The asset catalog is document-level because asset identity and backup integrity
must be global across workspaces. Entry references stay small, and future asset
metadata changes do not rewrite every referencing entry. Assets are immutable;
replacing an upload creates a new asset id.

The first image is the record cover image and the remaining images form its
gallery. Reordering the references changes the cover without changing binary
identity. Duplicate-entry behavior shares immutable asset references rather
than copying bytes.

```json
  "assets": [
    {

      "uri": "images/asset-moon-gate-7f2c.png",
      "originalFilename": "Moon Gate concept.png",
      "mediaType": "image/png",
      "byteSize": 842193,
      "sha256": "<lowercase-sha256>",
      "createdAt": "2026-07-09T18:30:00.000Z"
    }
  ],
  "worlds": [
    {
      "codex": {
        "places": [
          {

              {

                "caption": ""
              }
            ]
          }
        ]
      }

}
```

The omitted fields above are unchanged from the current document.

## ZIP Backup Contract

Recommended archive layout:

```text
valgaron-all-workspaces.zip
|-- valgaron-all-workspaces.json
`-- images/
    `-- asset-moon-gate-7f2c.png
```

For an active-workspace package, the JSON remains the existing single-workspace
backup shape and the ZIP contains only uploaded assets reachable from that
workspace. For a full package, it contains every uploaded asset reachable from
the current document. Unreferenced local files and recovery-snapshot-only files
are not included in a normal current-document backup.

"JSON as is" has a testable meaning: the ZIP builder receives the exact string
returned by the existing JSON serializer and writes those exact UTF-8 bytes to
the JSON entry. It does not rewrite asset URIs, reorder JSON, add a ZIP-only
manifest, or serialize a second time. The `images/...` URI in that JSON is the
matching relative path inside the archive.

Remote `https:` sources stay unchanged in JSON and do not appear under
`images/`. Fetching them would add network, copyright, hotlink, CORS, content
drift, and unbounded-file risks and would make export success depend on a third
party.

The package builder must fail closed when a referenced uploaded binary is
missing or its size/hash differs from metadata. It should report the affected
record/asset and leave JSON-only export available. It must never label a
partial ZIP as a complete backup.

## Decision Gates

No implementation phase that depends on a gate should start until the user
chooses an option or explicitly authorizes the recommendations.

### Gate 1: Records And Gallery Shape

Options:

1. Add `images` to every `WorldEntry` only, with the first image as cover and up
   to six images per record.
2. Add images to entries, workspaces, and in-fiction worlds in the first slice.
3. Add a single cover image only.

Recommendation: option 1. It fits the shared entry editor model and current
six-file uploader default while avoiding three distinct editor/mutation paths.
The model can later be reused for workspaces and in-fiction worlds.

Consequences:

- Web: one reusable entry media editor and gallery can serve all entry types.
- Mobile: one shared entry capability can be added after the web flow.
- Data: `WorldEntryBase.images` changes schema; workspace records stay stable.
- Tests: shared entry parsing, create, edit, duplicate, archive, and delete tests
  cover every entry category.
- Documentation: state clearly that workspace and planetary-world images are a
  later scope, not silently unsupported uploads.

### Gate 2: Uploaded URI And Asset Catalog Shape

Options:

1. Use generated relative `images/<asset-id>.<ext>` URIs plus a document-level
   asset catalog.
2. Use opaque `valgaron-asset://<id>` URIs plus a catalog and rewrite paths for
   ZIP export.
3. Store data URIs/base64 directly in each entry.

Recommendation: option 1. It preserves a plain URI string, makes extracted JSON
and image paths understandable, and avoids ZIP-time JSON mutation. Option 2 is
runtime-safe but makes the archive less transparent. Option 3 conflicts with
string-storage capacity and duplicates bytes.

Consequences:

- Web/mobile: renderers must resolve known `images/...` URIs through their
  platform asset repository before handing a source to the image component.
- Data: catalog/reference integrity becomes a schema validation rule.
- Tests: reject missing, duplicate, mismatched, unsafe, and orphan asset rows.
- Documentation: publish the relative-URI contract as part of schema version 4.

### Gate 3: Durable Binary Stores

Options:

1. Web IndexedDB for `Blob` values and app-managed mobile filesystem files.
2. Put image bytes in current `localStorage`/`AsyncStorage` JSON strings.
3. Hold uploads in memory until export.

Recommendation: option 1. The current string stores should continue to contain
only the typed document. Binary repositories use the stable asset id as the
key and expose the same small adapter contract.

Consequences:

- Web: introduce an IndexedDB repository, quota/error reporting, and object URL
  cleanup; no server is required.
- Mobile: add app-managed filesystem support and copy picker results immediately
  so expiring content-provider URIs never become durable state.
- Data: document save and binary save need a coordinator because they span two
  stores without a shared transaction.
- Tests: use an in-memory binary adapter for core/coordinator tests and focused
  IndexedDB/filesystem adapter tests.
- Documentation: update local-storage, clearing-data, uninstall, quota, and
  backup limitations.

### Gate 4: ZIP Export Only Or Round-Trip Backup

Options:

1. Deliver ZIP export and ZIP import together.
2. Deliver ZIP export first; keep imports JSON-only temporarily.
3. Treat ZIP as a media bundle only, not a restorable backup.

Recommendation: option 1. Once uploaded images are durable data, an export that
cannot restore them is not a complete portability story. JSON-only import must
remain for old and URI-only backups.

Consequences:

- Web: file input accepts `.json` and `.zip`; ZIP preview includes asset counts
  and total bytes before replacement confirmation.
- Mobile: ZIP import/export requires filesystem-backed temporary files and a
  document picker/share sheet instead of text-only sharing.
- Data: import validates JSON and all uploaded files before mutating either
  store; rollback removes newly staged files on failure.
- Tests: add hostile ZIP, missing file, checksum, replacement, rollback, and
  legacy JSON-only cases.
- Documentation: distinguish JSON reference exports from complete ZIP backups.

### Gate 5: Remote Image Packaging

Options:

1. Keep remote images as URI strings only.
2. Fetch and embed remote images during every ZIP export.
3. Offer an explicit future "make local copy" action that creates an uploaded
   asset; normal export still does not fetch remote sources.

Recommendation: option 1 for this slice, leaving option 3 as a later deliberate
feature. Export must be deterministic and must not generate unexpected network
requests.

Consequences:

- Web/mobile: remote images can fail offline and show a stable placeholder.
- Data: remote URI text remains portable but the remote bytes are not backed up.
- Tests: prove that the ZIP builder never asks the asset adapter for a remote
  URI and never performs network I/O.
- Documentation: disclose third-party requests when images render and warn that
  remote sources can change or disappear.

### Gate 6: Formats, Limits, And Byte Preservation

Options:

1. Preserve original bytes for JPEG, PNG, WebP, and GIF, with per-file, per-
   record, and per-package limits; reject SVG and undecodable/unknown content.
2. Transcode every upload to a standard raster format and strip metadata.
3. Accept every `image/*` browser-reported type without byte inspection.

Recommendation: option 1 for the prototype. It avoids quality loss and heavy
cross-platform processing. Validate magic bytes as well as MIME type and use a
conservative initial limit, proposed as 10 MB per file, six images per record,
and 100 MB of reachable uploaded bytes per ZIP. Exact numbers require product
approval and should be constants shared by validation and UI copy.

Consequences:

- Web/mobile: previews must report unsupported formats and size failures before
  editing the document. Animation behavior may differ by renderer.
- Data: original filename and MIME are metadata; generated URI extensions come
  from verified MIME.
- Tests: boundary values, MIME spoofing, image signatures, and aggregate export
  limits are required.
- Documentation: warn that preserved originals may retain EXIF metadata when a
  user intentionally shares a ZIP. Transcoding/metadata stripping remains a
  later privacy feature.

### Gate 7: Schema 3 Compatibility

Options:

1. Introduce schema `4`, read/migrate valid schema `3` documents by adding empty
   `images`/`assets`, and write only schema `4`.
2. Introduce schema `4` as another clean break with no schema `3` migration.
3. Add optional fields without changing the schema version.

Recommendation: option 1. Uploaded assets are a real durable model change, so
option 3 hides a compatibility break. A deterministic empty-array migration is
small and avoids discarding existing creative work.

Consequences:

- Web: read the v4 key first, then the v3 key through a migration path; never
  overwrite v3 until a v4 save succeeds.
- Mobile: mirror the same migration and save behavior for the mobile v4 key.
- Data: JSON import accepts schema 3 and normalizes it to schema 4; schema 4
  validates all image and asset invariants.
- Tests: fixtures for v3 migration, v4 parsing, storage-key precedence, failed
  v4 write, import preview, and recovery are mandatory.
- Documentation: update schema migrations, versioning, changelog, user guide,
  and service-worker upgrade checks.

### Gate 8: Web-First Versus Simultaneous Mobile Delivery

Options:

1. Land shared schema and web upload/render/ZIP round trip first, with mobile
   rejecting schema 4 until its asset implementation ships.
2. Land shared schema plus safe mobile schema-4 reading/render placeholders,
   then complete web before mobile upload/ZIP parity.
3. Hold release until web and native mobile have complete parity.

Recommendation: option 2. A shared document version must not make the companion
crash or silently discard image metadata. Web remains the first authoring
surface; mobile can initially preserve references and show explicit unavailable
states before gaining picker, filesystem, ZIP, and share-sheet parity.

Consequences:

- Web: can ship the complete first authoring workflow earlier.
- Mobile: must at minimum round-trip schema 4 without deleting image fields and
  must not claim a ZIP is complete before local binaries are available.
- Data: shared core ships first; platform capability status remains outside the
  durable world document.
- Tests: frontend parity fixtures must verify lossless schema round trips during
  the staged rollout.
- Documentation: release notes must identify the temporary mobile capability
  boundary and its backup implications.

## Component And Module Boundaries

### Shared Core

Add focused modules rather than putting binary APIs in `WorldDocument` parsing:

- `imageAssets.ts`
  - URI classification and normalization.
  - Asset/reference validation.
  - Referenced-asset selection for full and active-workspace exports.
  - Reference counting across current document and recovery snapshots.
  - Image mutation helpers that preserve immutable updates.
- `imageAssetLimits.ts`
  - Allowed MIME values and approved size/count limits.
- `codexDataPortability.ts`
  - Extend export options and preview counts.
  - Continue owning the canonical JSON string.
- `zipBackup.ts`
  - Platform-neutral package plan: JSON filename/text and ordered asset
    descriptors. Do not import DOM, IndexedDB, React Native, or filesystem APIs.
- `worldDocument.ts`
  - Parse schema 4, migrate schema 3, and enforce referential integrity.

The core must not contain `Blob`, browser object URLs, Expo file URIs, or ZIP
download/share behavior.

### Platform Asset Adapter

Use a narrow asynchronous contract in the platform package:

```ts
export type BinaryAsset = {
  bytes: Uint8Array;
  mediaType: string;
};

export type BinaryAssetRepository = {
  read(assetId: string): Promise<BinaryAsset | null>;
  write(assetId: string, asset: BinaryAsset): Promise<boolean>;
  remove(assetId: string): Promise<boolean>;
  listIds(): Promise<readonly string[]>;
};
```

If native memory pressure makes `Uint8Array` unsuitable, add a platform-specific
stream/file-handle export service without weakening the shared logical
contract. Do not force a fake synchronous abstraction across browser and native.

### Web

- Add an IndexedDB repository with one database/store version dedicated to
  immutable image blobs.
- Add an asset URL resolver that creates and caches object URLs only while a
  mounted consumer needs them, then revokes them.
- Replace/refactor the current temporary-URL uploader so `File` validation and
  staged asset persistence occur before the entry reference is committed.
- Add a ZIP download helper next to `fileDownloads.ts`, but keep archive
  assembly out of React components.
- Add an application save coordinator so binary writes succeed before the JSON
  document starts referencing them.
- Keep remote image rendering as normal `https:` image loading with failure and
  offline placeholders.

### Mobile

- Use a system picker, then immediately copy accepted content into an app-owned
  image directory keyed by asset id.
- Store asset bytes in files, not AsyncStorage. Keep schema metadata in the
  existing JSON persistence flow.
- Resolve uploaded asset URIs to app-owned file URIs only at render time.
- Write ZIPs to a temporary/cache path and invoke the native share/save sheet;
  clean temporary package files after completion when safe.
- Treat Android content URIs and iOS temporary picker URLs as input handles,
  never as durable JSON values.
- Add only SDK-compatible Expo dependencies after the dependency decision is
  reviewed; do not rely on transitive packages already present in the lockfile.

## Save, Delete, And Recovery Semantics

The two-store save sequence is:

1. Validate draft references and selected bytes.
2. Generate immutable asset ids, safe paths, metadata, and hashes.
3. Write new binaries to the platform repository.
4. Update the in-memory document with asset metadata and entry references.
5. On explicit web Save or mobile device save, persist the JSON document.
6. If JSON persistence fails, keep the prior saved document valid and mark new
   binaries as unreferenced candidates for cleanup; show a clear failure.

Removal from an entry removes only its reference. Remove current-document asset
metadata after its last current-document reference disappears. Physical binary
garbage collection must scan both the current document and retained recovery
snapshots because a snapshot may be the only remaining owner. When snapshot
retention evicts the last reference, the binary becomes collectable.

Reset, import replacement, entry deletion, workspace deletion, and snapshot
restore must use the same reachability rules. Never delete a binary merely
because one entry was deleted. Archive does not affect reachability. Duplicate
shares the immutable binary.

Run garbage collection after a successful save/snapshot mutation and at safe
startup maintenance points, not before a replacement transaction commits.
Report cleanup failures as diagnostics without invalidating an otherwise valid
document.

## ZIP Import Transaction

If Gate 4 accepts round-trip import:

1. Detect JSON versus ZIP by verified content, not filename alone.
2. Bound compressed size, expanded size, entry count, and path depth before
   extraction.
3. Reject absolute paths, `..`, backslashes as separators, duplicate paths,
   encrypted entries, links, unexpected executables, and multiple JSON roots.
4. Read and validate the JSON through the same schema/import path used today.
5. Build the exact expected uploaded path set from validated JSON.
6. Require every expected file once; reject or explicitly warn about unexpected
   files according to the approved import policy.
7. Verify MIME signature, byte size, and SHA-256 for each expected image.
8. Show the existing replacement preview plus uploaded image count and total
   bytes.
9. After confirmation, create the normal recovery snapshot, stage all new
   binaries, persist the new document, then garbage-collect old unreachable
   binaries.
10. If any stage fails, retain the prior document and binaries and remove only
    newly staged unreferenced files.

Do not render image content during import validation. ZIP contents are data,
not HTML or executable content.

## External Image Privacy And Security

- Allow persisted remote sources from `https:` only in production. A narrowly
  scoped localhost exception may exist in development tests without becoming
  accepted user data.
- Rendering a remote image sends a request to its host and can reveal normal
  request information such as IP address, user agent, and referrer behavior.
  State this near URI entry and in privacy documentation.
- Do not preflight/fetch a remote URI merely to save the record. Rendering is
  the network action.
- Do not proxy, cache, transform, or package remote images in this phase.
- Apply conservative referrer policy and image CSP rules on web where supported.
- Treat image decode errors as display failures, not application crashes.
- Reject SVG in the first slice because it expands script/external-resource and
  cross-renderer risk. Revisit only with an explicit sanitization design.
- Preserve original upload bytes only after magic-byte/type/size validation.
- Generated storage/ZIP paths prevent filename traversal and Unicode filename
  ambiguity.
- Never render imported captions, filenames, or URIs as HTML.

## Diagnostics And Failure States

Extend content-free diagnostics with counts only by default:

- referenced web image count;
- referenced uploaded image count;
- uploaded asset metadata count;
- locally available binary count;
- missing binary count;
- checksum/size mismatch count;
- unreachable binary count;
- reachable uploaded byte total;
- asset-store last error category, without filenames, URIs, ids, or hashes.

Expected user-facing failures include unsupported type, oversize file, quota
exceeded, remote load failure, missing local binary, checksum mismatch, ZIP too
large, ZIP assembly failure, canceled picker/share sheet, and failed cleanup.
Each error should say whether the JSON document remains safe and what recovery
action is available.

## Test Strategy

### Shared Core

- Schema 3 migrates losslessly to schema 4 with empty image/asset arrays.
- Remote and uploaded URI parsing accepts only the approved forms.
- Asset ids, URIs, paths, reference ids, and catalog rows are unique.
- Uploaded references require matching metadata; remote references do not.
- Active export includes only active-workspace-reachable assets.
- Full export includes all current-document-reachable assets exactly once.
- Duplicate shares assets; last-reference deletion updates metadata correctly.
- Snapshot-aware reachability retains and releases binaries correctly.
- Diagnostics expose counts, not content.

### Web

- Picker validation covers count, size, MIME, signature, and same-file reselection.
- Entry cancel/save/remove correctly handles staged objects.
- IndexedDB quota/read/write failures preserve the last valid JSON document.
- Object URLs are reused while mounted and revoked after last use.
- Remote image failure renders an accessible placeholder.
- ZIP JSON bytes exactly equal the canonical serializer output.
- Unzipped uploaded bytes, sizes, hashes, and paths match the asset catalog.
- ZIP creation performs no remote request.
- Import rejects traversal, bombs/limits, duplicates, missing files, spoofed
  types, mismatches, and invalid JSON before replacement.

### Mobile

- Temporary picker/content URIs are copied and never serialized.
- Missing local assets preserve JSON references with an explicit state.
- File write and share cancellation/failure leave the document valid.
- ZIP round trip matches core fixtures used by web.
- Android and iOS render/source adapters map the same logical URI correctly.

### Manual QA

- Web desktop, narrow mobile web, installed PWA, Android, and iOS where
  available.
- Reload/offline behavior for uploaded and remote images.
- Browser data clearing and mobile uninstall limitations.
- Save after quota exhaustion; recovery after corrupt metadata/missing binary.
- Active versus full package contents across two workspaces sharing an asset.
- Keyboard, focus, alt text, screen-reader labels, and touch targets.
- Large-but-valid package progress and memory behavior.

## Validation Gates During Implementation

After every source slice, run Prettier and the closest focused tests. Before a
phase is complete, run:

```text
npm test
npm run test:mobile
npm run typecheck
npm run typecheck:mobile
npx eslint .
npx vite build
```

Also run browser smoke checks after Data/editor route changes and the PWA check
after schema/storage/service-worker changes. Report Node/Vite compatibility
warnings separately from application failures. Do not start a verification dev
server without tracking and stopping it under repository process hygiene.

## Performance And Capacity Budget

- Never put binary bytes or base64 in React document state, textareas,
  localStorage, AsyncStorage, diagnostics, or recovery-snapshot JSON.
- Generate thumbnails with object/file URIs; do not decode every full image at
  once in a list.
- Hash and assemble asynchronously with visible progress.
- Prefer ordered reads and bounded concurrency over loading every asset
  simultaneously.
- If the selected ZIP codec cannot stream on a target, enforce the approved
  aggregate package limit before allocation and document the temporary limit.
- Add asset-count/byte dimensions to large-world performance fixtures without
  embedding large test binaries in the repository.

## Definition Of Done

- Accepted entry types can add, edit, reorder, remove, save, reload, and display
  HTTPS and uploaded raster images.
- Persisted JSON contains URI strings and metadata only; it never contains
  `blob:`, picker, temporary `file:`, or base64 payloads.
- Every uploaded reference resolves through one valid asset row and one locally
  stored binary, or produces an explicit diagnosable missing state.
- Full and active ZIPs contain the exact canonical JSON string and exactly the
  reachable uploaded binaries at their JSON URI paths.
- Remote images are never fetched by export and are documented as link-only.
- ZIP import, if approved, validates the entire package before replacing local
  state and can restore all uploaded images.
- Schema 3 compatibility follows the approved gate without silent data loss.
- Recovery snapshots and garbage collection cannot delete still-referenced
  binaries.
- Web and mobile preserve the shared schema throughout any staged delivery.
- Accessibility, privacy, security, capacity, and failure-state documentation is
  updated without making cloud, availability, or backup guarantees.
- Required focused, web, mobile, type, lint, build, PWA, and review tests pass,
  with environment-only warnings reported separately.

## Recommended Approval Set

For a coherent first implementation, approve Gate options `1, 1, 1, 1, 1, 1,
1, 2` respectively, along with the proposed initial limits of 10 MB per image,
six images per entry, and 100 MB of reachable uploaded bytes per ZIP. This keeps
the first authoring surface focused on entries, preserves current JSON behavior,
provides a real restorable backup, and stages mobile without allowing schema
loss.
